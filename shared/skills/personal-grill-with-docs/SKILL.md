設計・企画を容赦なく詰めながら、合意できたドメインモデル(用語集と重要な設計判断)を**その場で repo に書き残していく**スキルです。`personal-grill-me` の「容赦ないインタビュー」に、**ドメインモデルを成果物として育てる規律**を足したものです。

## このスキルを使う/使わないの境

- **使う**: 用語を固めたい、ユビキタス言語を整理したい、設計判断を ADR に残したい、ドメインモデルを育てながら設計を詰めたい ── と**明示**されたとき。
- **使わない(= personal-grill-me に委譲)**: 成果物を残さない素の壁打ち・方針出し。ドキュメント生成が要らないときにこのスキルを起動しない(二重発火を避ける)。

## インタビューの進め方(personal-grill-me と同じ核)

- **質問は一度に一つだけ。** まとめて投げない。
- **各質問に自分の推奨案と理由を添える。** 「どう思う?」で終わらせない。
- **コードベースで答えられることは聞かずに調べる。**
- **曖昧・思考停止・矛盾・希望的観測を潰す。** 「いい感じ」「業界標準だから」「たぶん大丈夫」で先に進ませない。
- **日本語で。**

## ドメインモデルを育てる(このスキル固有の規律)

設計を読んだり消費したりするだけでなく、**モデルを変えにいく**のがこのスキルです。用語と判断が固まった瞬間に書き留めます。

### 用語集と突き合わせる
ユーザーの語が既存の `CONTEXT.md` の語義と衝突したら、その場で指摘する。「glossary では『キャンセル』を X と定義しているが、いま言っているのは Y。どっち?」

### 曖昧な語を正準語に
あいまい/多義的な語には、precise な正準語を提案する。「『アカウント』は Customer のこと? User のこと? 別物だよ。」

### 具体シナリオで境界を詰める
関係を議論するときは、エッジケースのシナリオをこちらから作って境界の精度を上げる。

### コードと突き合わせる
「こう動く」と言われたら、コードがそれに同意しているか確認する。矛盾を見つけたら surface する。「コードは Order 全体をキャンセルしているが、今『部分キャンセルできる』と言った。どっちが正?」

### CONTEXT.md を inline 更新
用語が固まったら、その場で `CONTEXT.md` を更新する。後でまとめてやらない。書式は [CONTEXT-FORMAT.md](./CONTEXT-FORMAT.md)。`CONTEXT.md` は**用語集に徹し**、実装詳細・仕様・スクラッチを書かない。

### ADR は控えめに
ADR を提案するのは、次の **3 つすべてが真**のときだけ:

1. **覆しにくい** — 後で気が変わったときのコストが大きい
2. **文脈なしでは驚く** — 将来の読み手が「なぜこうした?」と思う
3. **実トレードオフの結果** — 本当に代替案があり、理由があって選んだ

どれか欠けたら ADR にしない。書式は [ADR-FORMAT.md](./ADR-FORMAT.md)。

## ファイル構成

ほとんどの repo は単一コンテキスト:

```
/
├── CONTEXT.md
├── docs/adr/0001-*.md
└── src/
```

root に `CONTEXT-MAP.md` があれば複数コンテキスト(map が各コンテキストの所在と関係を指す)。ファイルは**遅延作成**する ── 書くことができた最初の瞬間に作る。`CONTEXT.md` が無ければ最初の用語が固まったときに、`docs/adr/` が無ければ最初の ADR が要るときに作る。

## public-safe / 運用ルールとの関係

- 生成する `CONTEXT.md` / `docs/adr/` は **repo ローカルの開発者向け成果物**。secret / private な参照先 (URL・絶対パス) / PII を**書かない**。公開してよいか迷う内容は入れない。
- ドメインモデル(用語集・設計判断)と、プロジェクト運用の正本(計画・ログ・現在地)は**別層**。後者は別途のナレッジツールが正本で、このスキルはそこへ書かない。

## 終わり方

主要な分岐が解消されたら、決定事項・未解決の論点・**今回書いた/更新した CONTEXT.md と ADR** を簡潔にまとめて締める。だらだら続けない。
