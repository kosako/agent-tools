# ADR の書式

ADR は `docs/adr/` に置き、連番で命名する: `0001-slug.md`, `0002-slug.md` …。
`docs/adr/` は**遅延作成**する ── 最初の ADR が要るときに作る。

## テンプレート

```md
# {決定の短いタイトル}

{1〜3 文: 文脈は何で、何を決め、なぜそうしたか。}
```

これだけ。ADR は 1 段落でよい。価値は決定が「なされたこと」と「なぜか」を記録する点にあり、節を埋めることではない。

## 任意の節

genuine な価値があるときだけ足す。多くの ADR には不要:

- **Status** frontmatter(`proposed | accepted | deprecated | superseded by ADR-NNNN`)— 決定を見直すときに有用
- **Considered Options** — 却下した代替案を覚えておく価値があるときだけ
- **Consequences** — 非自明な下流影響を明示する必要があるときだけ

## 採番

`docs/adr/` の最大番号を探して +1 する。

## ADR を提案する基準

次の **3 つすべて**が真のときだけ:

1. **覆しにくい** — 後で気が変わったときのコストが大きい
2. **文脈なしでは驚く** — 将来の読み手がコードを見て「なぜこうした?」と思う
3. **実トレードオフの結果** — 本当に代替案があり、理由があって選んだ

覆しやすいなら省く(どうせ覆す)。驚かないなら誰も理由を問わない。代替が無かったなら「当然のことをした」以上に記録すべきものがない。

### 該当する例

- **アーキテクチャの形。** 「モノレポにする」「write model は event-sourced、read model は Postgres に projection」
- **コンテキスト間の統合パターン。** 「Ordering と Billing は同期 HTTP でなくドメインイベントで連携」
- **ロックインを伴う技術選定。** DB、メッセージバス、認証プロバイダ、デプロイ先。すべてのライブラリではなく、差し替えに四半期かかる類のものだけ。
- **境界・スコープの決定。** 「顧客データは Customer コンテキストが所有、他は ID 参照のみ」。明示的な no も yes と同じくらい価値がある。
- **あえて定石から外す決定。** 「ORM でなく手書き SQL にする、理由は X」。読み手が逆を仮定しそうなもの。次のエンジニアが「直そう」とするのを止める。
- **コードに見えない制約。** 「コンプライアンス上 AWS を使えない」「パートナー API 契約で応答 200ms 未満が必須」
- **非自明な却下理由。** GraphQL を検討して REST を選んだ微妙な理由があるなら記録する。さもないと半年後にまた提案される。
