# personal-repo-audit

リポジトリ全体を複数の観点で体系的に監査し、深刻度と `file:line` つきの所見に
まとめる手順です。1 つの視点で通読するのではなく、観点を分けて並列に調べ、結論だけを
集約することで、抜けの少ない監査を速く行います。

## なぜこう進めるのか

コードベースの問題は種類ごとに探し方が違います (バグの探し方と、設計乖離の探し方は
別)。1 人 (1 パス) で全部見ようとすると、どれかの観点が浅くなります。観点ごとに独立した
調査を並列に走らせ、各々が自分の観点に集中し、最後に束ねると、網羅性と速度が両立します。

## 手順

### 1. 監査範囲と観点を決める

まず対象範囲を確認します (リポジトリ全体か、特定の sub-tree か、直近の差分か)。
そのうえで観点を選びます。既定の観点:

- **correctness / bug**: 明確な不具合、境界条件、エラーハンドリングの欠落。
- **security**: 危険な入力処理、secret の混入、権限・サンドボックスの抜け。
- **設計 / 一貫性**: 重複、責務の混在、ドキュメントや宣言と実装の乖離。
- **テスト**: 観点の抜け、壊れやすい/無意味なテスト、カバーされていない経路。
- **ドキュメント陳腐化**: 実態とずれた記述、古い手順。
- **依存 / ビルド**: 不要・危険な依存、再現性。

対象の性質に合わない観点は外し、必要なら足します。

### 2. 観点ごとに並列で調べる

サブエージェントを並列に fan-out できる環境 (例: Agent ツール) があれば、観点ごとに
1 つずつ独立したサブエージェントを起動し、**各自その観点だけ**を調べさせ、結論
(所見のリスト) だけを返させます。並列化できない環境では、観点を 1 つずつ順に調べます。

各観点の調査には次を必ず含めさせます:

- 所見ごとに `file:line` を付ける (後で検証・修正できるように)。
- 深刻度を付ける (例: high / medium / low)。
- 推測と確証を分ける。

read-only を既定とします。監査の段階でコードを書き換えません。

### 3. 集約して重複を除く

各観点から返った所見を 1 つに束ね、同じ箇所・同じ問題の重複をまとめます。観点を
またいで重なる所見は最も深刻なものに寄せます。

### 4. (任意) 重要な所見を裏取りする

high の所見や確証の弱いものは、別の独立した視点で「本当に問題か」を検証してから
報告に残すと、誤検知 (plausible だが実は無害) を減らせます。確証が取れないものは
「未確証」と明示します。

### 5. 報告する

次の構造で報告します:

- **サマリ**: 全体の健全性と、最も重要な数点。
- **所見 (深刻度順)**: 各所見に `file:line`、何が問題か、なぜ問題か、推奨対応。
  high → medium → low の順。
- **対象範囲と限界**: 何を見て何を見ていないか (時間・範囲で絞った場合は明示)。

範囲を絞った/打ち切った場合は、それを黙って省かず明記します (見ていないものを
「問題なし」と誤読させないため)。

## やってはいけないこと

- 監査段階でコードを **書き換えない** (read-only。修正は別途、合意のうえ)。
- 所見を **でっち上げない**。確証の弱いものは「未確証」と書く。
- 範囲を絞ったのに「全部見た」かのように報告しない。
- 特定の参照先 / 絶対 path / secret を報告やコミットに焼き込まない。

## 例

**ユーザー**: 「このリポジトリ、一回ちゃんと棚卸ししたい」

**良い進め方 (この skill 適用後)**:

> 対象はリポジトリ全体ですね。6 観点 (bug / security / 設計 / テスト / docs / 依存) で
> 並列に監査します。
>
> (各観点を並列調査 → 集約 → high の裏取り)
>
> **サマリ**: 全体は健全。high は 2 件 (どちらも security)。…
> **所見 (深刻度順)**:
> - 🔴 high `src/auth.ts:42` — 入力の検証漏れ。理由…。推奨…。
> - 🟡 medium `lib/util.ts:88` — 重複ロジック。…
> **対象範囲と限界**: ソース全体を見た。生成物 / 大きな vendored dir は除外。

観点を分けて並列に調べ、所見に出典と深刻度を付け、範囲の限界まで明示しています。
